Pulled 5 new commits and went through schema, routes, seed data, and tests — this is a strong batch, with one precise scope distinction worth being exact about.

**`amount_usd` is real, not assumed.** I checked the schema directly: `doublePrecision("amount_usd")` on `position_records`, nullable, with an honest comment that legacy rows will be null. It's threaded through `createPosition.ts`, both export routes, and the harvest route consistently — nothing references a phantom field.

**The wash-sale detector is honestly scoped.** Its own docstring states plainly that IRC §1091 applies to stocks and securities, that the IRS hasn't officially extended it to crypto, and that this is a conservative practitioner marker, not a legal determination — then the API response repeats that same disclaimer to the end user. That's the same discipline as the earlier §6501 fix, now applied by default rather than something I had to catch.

**Real test coverage, not placeholder tests.** 38 test cases across three files. I sampled `washSale.test.ts` directly — it covers actual boundary conditions: exactly-30-days, date-diff symmetry, wallet mismatch, event-type mismatch, gain+gain (no pair), unknown-amount-treated-as-loss. That's meaningful coverage, not padding.

**Two concrete things I checked specifically because they're easy to get wrong, and both are right:** the harvest-candidates and review-queue routes are registered *before* `/positions/:id` — so the Express greedy-match bug their own docs warn about doesn't actually occur. And the new `/export/dossier` route calls the exact same `buildAuditPackageData`/`buildPatternReportData`/`buildCommentLetterData`/`buildCpaHandoffData` functions the four individual export routes already use — refactored into shared functions, not a fifth copy of the logic. Seed data matches too: 5 chains, 10 protocol rows (Aave V3 + Uniswap V3 × 5 chains), confirmed by direct count.

**The one thing worth being precise about, not a bug:** the harvest scanner queries positions already classified as `taxable_disposition` — meaning it surfaces losses you've *already realized* and flags wash-sale risk among them. That's genuinely useful. But it's not the forward-looking tool "loss harvesting scanner" usually implies — "which of my current, still-held positions are underwater right now and worth selling before year-end" — because that needs the lot-inventory model (current holdings, basis, unrealized gain/loss per wallet) I flagged a few turns back as still missing. It still doesn't exist. What got built is a smart, honestly-scoped way to get real value *without* that dependency — worth being clear it's realized-loss aggregation plus wash-sale flagging, not unrealized-position analysis, so it doesn't get described as the thing it isn't.

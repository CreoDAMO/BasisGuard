Here's the design doc summarized, then exactly what's built versus what's still just a plan.

**The doc proposes three layers:** (1) a chain/protocol hierarchy (chains table with L2→parent links, protocols scoped per chain), (2) an adapter/parser engine that reads raw on-chain events and turns them into Position Records automatically (Uniswap, Aave, Coinbase, Bridge adapters via a ProtocolRegistry), and (3) self-service onboarding so CPAs/partners can submit new chains and protocols for admin approval instead of everything being hardcoded.

**Layer 1 — built, and solid.** `chains.ts`, `protocols.ts`, and `submissions.ts` schemas all match the design closely — `is_l2`/`parent_chain_id` hierarchy, chain-scoped protocol contract addresses, submission tables with `status: pending/approved/rejected`. The approval flow in `submissions.ts` does exactly what the doc specifies: approving a chain submission resolves the parent chain by slug and auto-creates the real `chains` row; approving a protocol submission resolves the chain and auto-creates the `protocols` row. It's actually stricter than the doc in one place — `submitter_credential` is a required field on every submission, not just implied by role.

**Layer 3 — built, matches the doc.** `/submit/chain`, `/submit/protocol`, `/admin/submissions` with approve/reject — all present and wired to the auto-create behavior above.

**Layer 2 — not built at all.** I searched the whole repo for `adapter`, `registry`, `uniswap`, `aave` — zero matches outside the doc itself. `ProtocolRegistry`, `BaseProtocolAdapter`, and every named adapter (Uniswap, Aave, Coinbase, Bridge) don't exist yet. That's the part that would actually read a transaction and auto-generate a Position Record with the right chain/protocol context — right now chain data can be *stored*, but nothing *populates* it from real activity. Worth flagging too: the doc's adapter skeleton is written in Python/SQLAlchemy, but the real stack is TypeScript/Drizzle/Express — so this isn't a "port the skeleton" job, it needs a native rewrite when you get to it.

**A gap the doc assumed away:** it says `ALTER TABLE wallets ADD COLUMN chain_id` and the same for `transactions` — but neither table exists in this codebase. `position_records.wallet_id` is a plain text field, not a foreign key to a real wallets table. So wallet-bound chain scoping and the cross-chain wallet-matching idea in the doc don't have anything to attach to yet.

**The one I'd actually prioritize before any of the above: there's no authentication anywhere.** `app.ts` has no auth middleware at all — CORS is wide open, and every route including `PATCH /admin/submissions/chain/:id/approve` and position sign-off is unauthenticated. The doc's own "Roles & Permissions" section (Super Admin / CPA-Partner / Reviewer) and its stated security control ("submissions require verified CPA license") aren't implemented — anyone who can reach the API can approve a chain submission or sign a position off under any `reviewer_name`/`credential` they type in. That's a bigger problem than the missing adapters, because it undermines the evidentiary weight of the sign-offs you've already built — an attestation only means something if not just anyone can produce one.
